Overview

SecureFlow is a web-based security scanning tool built with Flask. It combines static code analysis (SAST via Semgrep), dynamic application testing (DAST via OWASP ZAP), and HTTP security header inspection into a single interface.

The frontend lives in frontend/ and serves three pages: a scanner, a reports list, and this documentation. All scan results are rendered server-side using Jinja2 templates and stored as .txt report files in the project root.

Note: The templates use Jinja2 syntax ({% ... %}, {{ ... }}) and will only render correctly when served through Flask at http://127.0.0.1:5000 — not when opened directly as a static file.

Running the app

Start the Flask development server from the project root.

From the project root

cd SecureFlow
python frontend/app.py

The app starts on http://127.0.0.1:5000 with debug mode enabled.

With a virtual environment

cd SecureFlow/frontend
source myenv/bin/activate       # macOS / Linux
myenv\Scripts\activate          # Windows

python app.py
Working directory matters. Always run from the project root so that REPORTS_DIR resolves correctly and report files are saved alongside existing report_*.txt files.

Prerequisites

The following tools must be installed and accessible on your PATH for scans to run.

Python 3.9+ Runtime for Flask and all backend modules
Flask Web framework — pip install flask
Semgrep SAST engine — pip install semgrep
OWASP ZAP DAST engine — must be running with API access enabled
requests HTTP client for header checks — pip install requests

Routes

The app exposes three URL routes, all defined in frontend/app.py.

GET / Renders the scanner form with no results
POST / Runs the scan, saves a report, and re-renders with results
GET /reports Lists all report_*.txt files sorted newest first
GET /reports/<filename> Shows summary stats and full content of one report
GET /docs This documentation page

The /reports/<filename> route validates the filename against report_\d{{8}}_\d{{6}}\.txt before serving, returning 404 for anything that does not match.

Templates

All templates live in frontend/templates/ and share the same Inter font, colour variables, and header component.

index.html Scanner form and inline scan results. Receives the result context variable after a POST.
reports.html Report list table. Receives a reports list — each item has filename, datetime, and meta.
report_detail.html Single report view. Receives filename, content (raw text), and meta (parsed counts).
docs.html This page. Static — receives no context variables from Flask.

Scan form

The scanner form on / submits three fields via POST.

code_path Absolute path to a local codebase. Used when scan type is sast or both. Ignored if empty.
url Target URL for DAST and header checks. http:// is prepended automatically if missing. Used when scan type is dast or both.
scan_type One of sast, dast, or both. Controls which backend modules are invoked.

Scan types

Each scan type maps to one or more backend modules.

SAST — Static Analysis

Runs backend/semgrep_scan.py, which invokes Semgrep with the p/python ruleset against the provided code path. Results are a list of {{rule_id, message, severity}} dicts. Semgrep severity values (ERROR, WARNING) are normalised to HIGH / MEDIUM / LOW by generate_report_data in backend/report.py.

DAST — Dynamic Analysis

Runs backend/zap_scan.py against the live target URL and returns a path to the ZAP HTML report. Also runs backend/headers_check.py, which fetches the URL and checks response headers against a rule set loaded from backend/headers_rules.json.

Both

Runs SAST on code_path and DAST + header checks on url in sequence. Both inputs must be provided.

Result object

After a POST, app.py passes a result dict to index.html. Here is the full shape:

result — Python dict
{
  "summary": {
    "total":  int,      # combined finding count
    "high":   int,
    "medium": int,
    "low":    int
  },

  "sast": [             # present only if SAST was run
    {
      "severity": "HIGH" | "MEDIUM" | "LOW",
      "message":  str
    }
  ],

  "dast": {             # always present
    "status":      "Completed",
    "summary":     str,
    "report_path": str  # absolute path to ZAP HTML report
  },

  "headers": {          # present only if DAST was run
    "findings": [
      {
        "header":         str,   # e.g. "Content-Security-Policy"
        "severity":       str,   # "HIGH" | "MEDIUM" | "LOW"
        "category":       str,
        "risk":           str,   # description of the risk
        "recommendation": str    # fix suggestion
      }
    ],
    "analysis":     [str],
    "status_code":  int,
    "headers":      dict,        # raw response headers
    "url":          str
  }
}

The template accesses result.dast.report_path for the ZAP report link and h.recommendation (not h.fix) for header fix suggestions.

Severity levels

Severity is normalised across SAST and header findings before being passed to templates.

HIGH Semgrep ERROR Critical vulnerabilities requiring immediate attention
MEDIUM Semgrep WARNING Notable issues that should be addressed soon
LOW Semgrep INFO Minor findings or best-practice recommendations
ERROR Raw Semgrep output Shown as-is before normalisation in some legacy paths
INFO Header rules fallback Informational findings with no direct exploit risk

How reports work

Every scan automatically writes two output files to the project root.

Text report

Generated by backend/report.py → generate_text_report(). Saved as report_YYYYMMDD_HHMMSS.txt in REPORTS_DIR (the project root). The /reports route scans for these files with glob("report_*.txt") and parses their headers for metadata.

Dashboard HTML

Generated by backend/dashboard.py → generate_dashboard(). Saved as dashboard.html in the project root, overwriting the previous one on each scan. It is a self-contained HTML file that can be opened directly in a browser without Flask.

REPORTS_DIR is resolved at startup as os.path.dirname(os.path.dirname(os.path.abspath(__file__))) — one level above frontend/, which is the SecureFlow project root.

Report format

Text reports follow a fixed plain-text structure that _parse_report_meta() reads to extract summary data for the reports list.

report_YYYYMMDD_HHMMSS.txt
=== SecureFlow Security Report ===

Generated At: 20260417_163437
Code Path:    /path/to/project
Target URL:   https://example.com

=== Executive Summary ===
Total Issues: 9
HIGH:   2
MEDIUM: 1
LOW:    6

=== SAST Findings (Semgrep) ===
[HIGH] Found 'subprocess' with shell=True ...

=== DAST Summary (OWASP ZAP) ===
OWASP ZAP baseline scan executed.
Detailed Report: /path/to/zap_report.html

=== Recommendations ===
- Fix high/medium issues first.
- Enforce secure headers and HTTPS.
- Integrate scans into CI/CD.

=== End of Report ===

The parser reads lines sequentially and stops once all six keys (target, code_path, total, high, medium, low) are found, making it efficient even for large report files.
